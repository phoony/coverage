/ Longer Explanation

What is an API Contract?

An API contract, such as OpenAPI outlines the structure and behavior of APIs. It specifies the available operations, their locations, and the required parameters. When properly defined, it allows consumers to understand and interact with a remote service without needing to know its internal workings. These descriptions can automatically generate documentation and server or client stubs.

What is API Testing?

Often APIs come with their own test suite which focuses on testing the APIs behavior. The developers could for example write a simple test client that calls certain API functions via HTTP and verifies that the server responds as expected. This is mostly testing the domain logic but can also test different behaviors such as error handling or performance.

The Goal

The execution of the API tests generates valuable traffic data. The traffic contains information about which endpoints are called, with which parameters and how often. This data can be used to generate statistics that show how thoroughly the API is tested. The best thing is: this is completely language agnostic since it only relies on the raw HTTP traffic. The goal of this work is to create a tool that does exactly that: analyze the traffic data and generate a report that shows how well the API is covered. You can help by participating in the survey mentioned above.

What is API Coverage?

API Coverage is a collection of metrics that measures the degree to which an API is tested. These metrics can vary depending on their definition and purpose. The important thing is that they give an idea of how well the API is tested and where the gaps are. The goal is that the generated coverage data can be used to help developers improve their test suite.

A desirable outcome would be to find a positive correlation between API Test Coverage and the robustness of the API. Equally desirable would be a correlation between API Test Coverage and found bugs in the API implementation. This could be used to predict the quality of the API based on the test coverage.

API Coverage Report

Measuring API coverage not only as a percentage but also providing insights into areas needing improvement is beneficial. A graphical representation can offer a quick overview of individual metrics and other relevant information, showing developers on where they should focus their efforts in the test suite.

Example Metrics

Note: Most of these metrics are taken directly from or inspired by the article Test coverage criteria for RESTful web APIs by Alberto Martin-Lopez, Sergio Segura, Antonio Ruiz-Cort√©s

Metric Name Short Description
Path Coverage

Every OpenAPI document defines a set of paths. How many of these paths are we testing?
Operation Coverage

Paths offer multiple operations which are accessed via the corresponding HTTP-Method (GET, POST, ...). Which operations are being tested?
Parameter Coverage

Operations can accept parameters. Are all of these Parameters used?
Parameter Value Coverage

Which values are being passed as parameters?

  • Are we testing with valid values?
  • Are we testing with multiple different values?
  • Are we testing edge cases?
Operation Flow Coverage

We can define specific sequences of operations that we name operation flows. Which of these operation flows are tested? Which are desirable to cover?
Satus Code Coverage

Can we observe all the possible status codes returned by operations?
Body Property Coverage

Responses are mostly encoded in JSON. They can also be defined in the API Contract. Can we observer all possible properties of the response objects?
